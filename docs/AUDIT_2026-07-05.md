## 1. Security & access — **mostly solid**

Verified good:
- **Structural-write authorization is centralised and sound** (`lib/api/auth.ts#isSpaceOwner`):
  claimed spaces require Clerk owner/editor via `getProjectAccess`; anonymous
  drafts require an exact ≥16-char `anon_owner_token` match (no "any non-empty
  token" shortcut). The NO-AUTH widget/style routes all call it. No IDOR.
- **Reads are role-gated**: `/spaces/[id]`, `/versions/[version]` both go
  through `authorizeSpaceViewer` (`spaceRead.ts`) → owner / share-link / member,
  returns null otherwise. No cross-tenant read via guessed IDs.
- **Private-read lockdown live** (migration 025): anon `SELECT` on
  `profiles`/`spaces`/`module_state`/`space_versions` → `42501`.
- **Admin routes gated** via `requireAdmin()` (`lib/admin`).
- **Security headers present** on prod: HSTS (1y, includeSubDomains), X-Frame
  DENY, nosniff, referrer-policy, permissions-policy.
- **Secrets server-side**: service-role/OpenAI/Clerk-secret only in `lib/server/*`
  and API routes; no client bundle leak.
- **Rate limits** present on the cost/AI routes (`/spaces`, `/clarify`,
  `regenerate`, `/upload`, `/projects`).

Findings:
- **MEDIUM — `/api/geocode` has no rate limit.** Public GET proxy to Photon;
  abusable as a free geocoding proxy / function-invocation DoS. Fix: per-IP
  persistent limit via the existing `take_rate_limit` RPC.
- **LOW — AI modules lack `import "server-only"`.** `lib/server/{classify,
  clarify,regenerate,contractDraft}.ts` read `OPENAI_API_KEY` but have no
  server-only guard (they are only imported server-side today; the guard is
  defense-in-depth against an accidental client import).

## 2. Data, DB & contract — **one blocker**

- **BLOCKER — Migration 018 (`spaces.modules_rev`) is MISSING in prod**
  (`spaces?select=modules_rev` → `42703`). Consequence, verified in code
  (`widgets/route.ts#persistModules`): the optimistic-concurrency guard
  (`.eq("modules_rev", expectedRev)`) is skipped on the missing-column
  fallback, so **concurrent structural edits silently overwrite each other**
  (no 409), and the **claim/slot unique index** from 018 is absent. Apply
  `supabase/migrations/018_modules_rev_claim_guard.sql` (idempotent), then
  re-probe (must return `42501`, not `42703`).
- Verified present: migrations 002–017, 019–027 markers all resolve; the frozen
  data-contract guard (`lib/contract.ts`) still compiles.
- **MEDIUM — `module_state` reindex on reorder/delete is still multi-statement**
  (known, `#5` in the open list): a mid-operation outage can leave state on the
  wrong index. Move to a service-role RPC.
